# website-check を並行確認にした理由

OpsKitPro の中でも、website-check はかなり重要なモジュールです。  
このページは、サイトの状態をまとめて確認するための診断画面ですが、実装していく中で「長い順番待ち」より「並行で見る」ほうが自然だと分かりました。

今回は、その理由を書きます。

## まず何を見たいのか

サイトが開かないとき、確認したいのは主に次の 4 つです。

- DNS
- HTTP
- SSL
- CDN

普通はこれらを順番に見ていきますが、実際の障害対応では、全部を同時に気にしています。

## 逐次フローの弱さ

よくある設計は、次のような流れです。

1. DNS
2. HTTP
3. SSL
4. CDN
5. 結果まとめ

この作りだと、ユーザーは「次のステップ」を待つことになります。  
でも、障害対応で欲しいのはステップの進捗ではなく、すぐ使える判断です。

## 並行確認に変えた理由

website-check では、複数の信号を同時に見て、最後にまとめる形に変えました。

### 1. 待ち時間が減る
複数の確認を並行で進められるので、ユーザーの体感が軽くなります。

### 2. 結論が早い
先に要点を出して、詳細は必要なときだけ開けるようにしています。

### 3. 部分成功を扱いやすい
外部サービスが一部失敗しても、見えている情報だけ先に出せます。

## 実装で意識したこと

### 1. 入力を正規化する
URL、ドメイン、パス付き URL が混ざると扱いがブレるので、先にホスト名へ正規化しています。

### 2. API 契約を揃える
フロントエンドとバックエンドで意味がズレないように、診断結果の構造を揃えています。

### 3. 詳細は折りたたむ
最初から JSON を全部見せるのではなく、結論を先に置いています。

## まとめ

website-check は、単なる「サイト確認ページ」ではなく、  
DNS / HTTP / SSL / CDN をまとめて見るための、障害対応向けの入口です。

逐次より並行確認のほうが、現場では自然でした。  
その結果、ページも実装もずいぶん扱いやすくなりました。
