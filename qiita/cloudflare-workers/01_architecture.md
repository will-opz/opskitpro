# OpsKitPro の配信基盤を Cloudflare Workers にした理由

OpsKitPro は、Cloudflare Workers 上で動かしています。  
これは流行りに乗ったというより、プロダクトの性質にかなり合っていたからです。

## 何を大事にしたかったか

OpsKitPro は、診断系のツールをすばやく返すサイトです。  
そのため、次の 3 点をかなり重視しました。

- 体感が軽いこと
- 配信がシンプルであること
- 運用しながら壊れにくいこと

特に運用ツールは、派手さよりも「すぐ開ける」「すぐ結果が出る」ことが重要です。

## Workers と相性が良かった点

### 1. Edge に近い
ユーザーから見た応答が軽くなりやすく、日本からの体感も安定しやすいです。

### 2. 配信とルーティングをまとめやすい
OpsKitPro は多言語ルーティングや canonical 管理があるので、エッジでルールを寄せられるのは便利でした。

### 3. 静的資産と API を同じ思想で扱える
診断系の API と、ブログや静的アセットの配信を同じ基盤に寄せられるのは大きいです。

## 選んだときの判断基準

Cloudflare Workers を選ぶとき、単に速いかどうかではなく、次の観点で判断しました。

- Next.js の運用に無理がないか
- 画像や静的ファイルが扱えるか
- 多言語や OGP のメタデータが破綻しないか
- デプロイが毎回重くならないか

結果として、OpsKitPro のような軽量な運用サイトには十分に合っていました。

## まとめ

OpsKitPro にとって Workers は、「配信先」以上の意味があります。  
サイトの軽さ、ルーティングの整理、運用しやすさをまとめて満たせる基盤だったので、自然にこの構成になりました。
